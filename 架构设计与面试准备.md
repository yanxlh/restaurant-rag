# 餐厅评价 RAG 系统 — 架构设计与面试准备

---

## 一、系统整体设计

### 这是什么系统？

用户可以用自然语言向指定餐厅提问（比如"这家店辣吗？""停车方便吗？"），系统从真实用户评价中检索相关内容，交给本地大语言模型生成回答，实时流式返回到界面。

这类系统叫 **RAG（Retrieval-Augmented Generation，检索增强生成）**。核心思路是：LLM 本身不存数据，所有领域知识来自实时检索。

### 数据流全链路

```
用户提问
  │
  ▼
① Redis 查缓存  ──命中──▶  直接返回（跳过后续）
  │未命中
  ▼
② 问题向量化（nomic-embed-text via Ollama）
  │
  ▼
③ ChromaDB 向量相似度搜索（按 restaurant_id 过滤，取 Top-5）
  │
  ▼
④ 拼接评价上下文 + 用户问题 → 构造 Prompt
  │
  ▼
⑤ qwen2.5:7b（via Ollama）流式生成回答
  │
  ▼
⑥ SSE 实时推送到前端      同时写入 Redis 缓存
```

## 二、为什么这样设计？各模块的设计动机

### 2.1 为什么用 RAG 而不是直接用 LLM？

LLM 只有训练数据截止日期之前的知识，不知道你系统里具体餐厅的评价内容。有两种选择：

| 方案 | 缺点 |
|---|---|
| Fine-tuning（微调） | 数据更新时要重新训练，成本高，延迟高 |
| RAG（检索增强） | 数据实时检索，无需重训，成本低 |

RAG 适合**内容频繁更新、需要来源可溯**的场景，餐厅评价完全符合。

### 2.2 为什么用向量数据库存评价？

评价是非结构化文本，用户提问语义各异，无法用 SQL `LIKE` 匹配。向量数据库把文本转成高维向量，通过**余弦相似度**找语义最接近的评价，这是语义搜索的标准做法。

### 2.3 为什么有两个数据库（MySQL + ChromaDB）？

这是**职责分离**原则：

- **MySQL**：存餐厅结构化信息（名称、地址、评分、营业时间），适合精确查询和关联操作
- **ChromaDB**：存评价向量，适合模糊语义搜索

把两者混在一起是错误的——关系型数据库不擅长向量运算，向量数据库不擅长 JOIN 查询。

### 2.4 为什么用 Redis 做缓存？

向量检索 + LLM 推理是整个链路中最慢的部分（本地 7B 模型单次响应 3-10 秒）。同一个问题被反复问时，没有必要重复计算。

缓存 Key 设计为 `{restaurantId}:{md5(question)}`，按餐厅隔离，防止不同餐厅的相同问题互相污染结果。

代码里还做了两个额外处理：
- **TTL 随机抖动 ±10%**（3240~3960 秒）：防止大量缓存同时过期导致的"缓存雪崩"
- **空值哨兵**（写入 `"NULL"` 字符串，TTL 5 分钟）：防止反复查询不存在的 key 导致的"缓存穿透"

### 2.5 为什么用 SSE 而不是 WebSocket？

LLM 生成是单向流——服务端生成 token 推给客户端，客户端不需要反向发消息。

| 协议 | 适用场景 | 本项目 |
|---|---|---|
| WebSocket | 双向实时通信（聊天室、游戏） | 过度设计 |
| SSE（Server-Sent Events） | 服务端单向推流 | 刚好合适 |

代码里用 Spring WebFlux 的 `Flux<String>` 配合 `MediaType.TEXT_EVENT_STREAM_VALUE` 实现，天然支持背压（Backpressure）。

前端没有用 `EventSource`，因为它不支持 POST 请求，改用 `fetch + ReadableStream` 手动读流。

### 2.6 为什么数据预处理要分三步（clean → chunk → ingest）？

- **clean**：Yelp 原始数据有噪声（HTML 实体、极短评价、空字段），清洗后质量更高
- **chunk**：长评价超过模型上下文窗口，切成小段后每段向量化更准确，检索粒度更细
- **ingest**：批量写入 ChromaDB（每批 50 条），避免单次请求体过大导致 OOM 或超时

## 三、技术选型：为什么选这个而不选那个？

### 3.1 LLM：Ollama + qwen2.5:7b vs OpenAI API

| 维度 | Ollama 本地 | OpenAI API |
|---|---|---|
| 费用 | 免费 | 按 token 计费 |
| 隐私 | 数据不出本机 | 数据发往第三方 |
| 延迟 | 取决于本机 GPU/CPU | 稳定低延迟 |
| 可用性 | 不依赖网络 | 需要网络和账户 |
| 模型质量 | 7B 参数，中文尚可 | GPT-4 更强 |

选 Ollama 的核心原因：**开发阶段零成本，且 Yelp 餐厅数据属于私有数据集，不适合发给第三方**。生产环境如果有预算可以换 OpenAI，Spring AI 只需改配置，代码不变。

为什么选 `qwen2.5:7b` 而不是 `llama3` 或 `mistral`？qwen2.5 是阿里开源的，**中文理解能力显著优于** Meta 和 Mistral 的模型，对中文回答质量更好。

### 3.2 向量数据库：ChromaDB vs Pinecone / Milvus / pgvector

| 方案 | 优点 | 缺点 |
|---|---|---|
| **ChromaDB** | 轻量、本地部署、开箱即用 | 单机、不支持分布式 |
| Pinecone | 全托管、生产级 | 收费、数据在云端 |
| Milvus | 分布式、高性能 | 部署复杂、重量级 |
| pgvector | 复用已有 PostgreSQL | 向量性能弱于专用库 |

开发/演示项目选 ChromaDB：Docker 一键启动，无需注册账号，与 Spring AI 集成成本最低。

### 3.3 后端框架：Spring Boot + Spring AI vs FastAPI + LangChain

| 维度 | Spring Boot + Spring AI | FastAPI + LangChain |
|---|---|---|
| 生态成熟度 | Spring 生态完整 | Python AI 生态更丰富 |
| AI 框架 | Spring AI（相对新） | LangChain（工具链更全） |
| 性能 | WebFlux 异步非阻塞 | asyncio 异步 |
| 企业适用性 | 更适合 Java 技术栈团队 | 更适合 AI/ML 团队 |

选 Spring Boot 的原因：这是一个后端工程项目，主要展示 Java 技术能力。Spring AI 1.0.0 已 GA，与 Spring 生态（事务、缓存、JPA）无缝集成。

### 3.4 前端：Vue 3 vs React

两者都可以。选 Vue 3 的原因是 Composition API 写流式 SSE 接收逻辑更简洁，并且 Vue 3 + Vite 启动极快，适合快速开发演示界面。

## 四、系统存在的问题与局限

### 4.1 向量维度与集合初始化问题
ChromaDB 集合在第一次插入向量时才确定维度。`nomic-embed-text` 输出 768 维，如果后续换了嵌入模型（维度不同），需要重建集合。

### 4.2 Spring AI 与 ChromaDB 版本兼容性
Spring AI 1.0.0 调用 ChromaDB v2 API，但集合不存在时错误消息格式不匹配，`initialize-schema: true` 无法自动初始化。需要在启动前手动预创建租户/数据库/集合（已记录在错误报告中）。

### 4.3 Ollama 本地推理延迟
7B 模型在 CPU 上首 token 延迟约 3-8 秒，流式输出后续每 token 约 50-200ms。没有 GPU 的机器体验较差。生产环境建议换成 API 服务或配备独立 GPU。

### 4.4 缓存 Key 用 MD5(question)，大小写和空格敏感度弱
代码已做 `toLowerCase().trim()`，但同义不同词的问题（"好吃吗" vs "味道怎么样"）会 miss 缓存，属于语义缓存问题，需要额外的相似问题聚合才能解决。

### 4.5 无身份验证
当前所有 API 无鉴权，生产环境需要加 JWT 或 Spring Security。

## 五、面试常见问题 & 参考回答

### Q1：什么是 RAG？它解决了什么问题？

RAG 是检索增强生成。LLM 有知识截止日期，无法知道实时或私有数据。RAG 的做法是：先从知识库检索与问题相关的文档片段，再把这些片段作为上下文交给 LLM 生成回答。这样 LLM 就能"看到"它训练时不知道的数据，且回答有据可查，幻觉（Hallucination）更少。

### Q2：向量相似度搜索的原理是什么？

文本先通过 Embedding 模型转成高维向量（本项目 768 维）。"意思相近的文本，向量在空间中距离更近"。搜索时把用户问题也向量化，在向量空间找距离最近的 Top-K 个文档。常用距离度量有余弦相似度（角度）、欧氏距离、内积，本项目用余弦相似度，对向量长度不敏感，效果更稳定。

### Q3：为什么 Redis 缓存 Key 要用 MD5？

两个原因：
1. 问题文本可能很长，直接用文本做 key 占用内存多，且 Redis key 有长度建议上限
2. MD5 输出定长 32 位十六进制，key 结构整齐，便于管理

注意：这里 MD5 不用于安全目的（不需要防碰撞），只用作固定长度哈希，所以用 MD5 完全合理。

### Q4：什么是缓存雪崩？你们怎么处理的？

缓存雪崩是指大量缓存同时过期，请求全部打到数据库（或在本项目中打到 LLM），造成瞬时过载。本项目的处理是给 TTL 加随机抖动（±10%），让缓存过期时间分散在一个区间内，不会同时失效。

### Q5：什么是缓存穿透？你们怎么处理的？

缓存穿透是指查询一个永远不存在的 key，每次都 miss 缓存，每次都触发后端查询，可以被用来恶意攻击。本项目用空值哨兵处理：当查询结果为空时，也向 Redis 写入一个 `"NULL"` 标记，TTL 设为 5 分钟。下次同样的请求命中缓存，返回空，不再查后端。

### Q6：SSE 和 WebSocket 的区别？什么时候选哪个？

| | SSE | WebSocket |
|---|---|---|
| 方向 | 服务端 → 客户端 单向 | 双向 |
| 协议 | 基于 HTTP | 独立协议，需升级握手 |
| 断线重连 | 浏览器自动重连 | 需手动实现 |
| 使用场景 | 日志推送、AI 流式输出、股票行情 | 在线聊天、游戏、协作编辑 |

LLM 流式回答是单向推送，用 SSE 最合适，WebSocket 是过度设计。

### Q7：Spring WebFlux 和 Spring MVC 有什么区别？为什么这里用 WebFlux？

Spring MVC 是同步阻塞模型，每个请求占用一个线程，等待 IO 时线程阻塞。Spring WebFlux 基于 Reactor，事件驱动、非阻塞，少量线程处理大量并发。

本项目用 WebFlux 的原因：LLM 推理时间长（秒级），如果用 MVC，每个问答请求会长时间占用一个线程，并发稍高就会耗尽线程池。WebFlux 在等待 Ollama 返回 token 的过程中不阻塞线程，可以同时处理更多请求。

### Q8：向量数据库里的 `filterExpression` 是做什么的？

相似度搜索默认在整个集合里找，但本项目一个集合存了所有餐厅的评价。如果不过滤，用户问 A 餐厅，可能搜到 B 餐厅的评价。`filterExpression(b.eq("restaurant_id", restaurantId))` 相当于 SQL 的 `WHERE restaurant_id = ?`，把搜索范围限制在当前餐厅的评价里。这是向量数据库的**元数据过滤（Metadata Filtering）**功能。

### Q9：Embedding 模型和 LLM 的区别？

| | Embedding 模型 | LLM（生成模型）|
|---|---|---|
| 输出 | 定长向量（如 768 维浮点数） | 文本（tokens）|
| 用途 | 语义相似度计算、向量检索 | 文本生成、问答 |
| 本项目 | nomic-embed-text | qwen2.5:7b |

Embedding 模型把文本压缩成一个"语义坐标"，不生成文字；LLM 根据上下文生成下一个 token。两者分工不同，缺一不可。

### Q10：如果系统要支持百万级餐厅评价，你会怎么优化？

1. **ChromaDB 换 Milvus 或 Pinecone**：支持分布式部署和更大数据量
2. **Ollama 换 vLLM 或 OpenAI API**：本地 7B 吞吐量有限，需要专业推理服务
3. **Redis 集群**：单节点 Redis 容量有限，改用 Redis Cluster
4. **向量索引调优**：ChromaDB 默认用 HNSW 索引，调整 `ef_construction` 和 `M` 参数平衡精度与速度
5. **评价分批异步写入**：用消息队列（Kafka/RabbitMQ）异步处理 ingest，避免阻塞主流程
6. **缓存语义聚合**：相似问题复用缓存，而不仅仅是完全相同的问题

## 六、一句话总结整个架构

> 用 Python 离线管道清洗 Yelp 数据并向量化写入 ChromaDB，Spring Boot 后端在用户提问时做向量检索 + Redis 缓存 + Ollama 本地大模型流式回答，Vue 3 前端通过 SSE 实时渲染，整个系统完全本地运行，无需任何云 API。
